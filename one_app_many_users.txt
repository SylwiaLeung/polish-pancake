1. To make an application available for multiple users, departments and companies, while ensuring data protection, one can architect a multi-tenant application. It is in some cases much more useful than installing separate instances of the application for each customer, in a single-tenancy.
There are many advantages of multi-tenancy; among those: 
- it reduces cost by sharing same power resources and software across its clients. The software requires licenses and support, and the cost for this will increase if you need to buy them for each customer, as in the case of single-tenancy. 
- in a multi-tenant application, you’ll only have to monitor and provide administration for a single platform, unlike single-tenant applications where you have to manage different sets for each client. This way more efficient and effective support can be delivered, including problem resolution and troubleshooting.
- multi-tenant architecture will make it easy for you to increase your capacity when more power is required. The total capacity of the entire system increases and becomes more scalable when you bring in new hardware to the platform. Adding features for a single client makes it available to other clients.
- upgrading the software version or system resources for is easy since there is a single, centralized place for upgrading, installation etc.
Obviously, there are also disadvantages for using multi-tenancy. If you decide on chosing it, you might be faced with following issues: a multi-tenant app has lesser flexibility than a single-tenant app to set low-level configurations. It is more complex than a single-tenant app, whose configuration can basically remain static. In a single-tenant application, you don’t require any code to know which tenant a web request is intended for or to protect your clients from the leakage of data.
Also, multi-tenant has less areas prone to failure, but those failures can be much more serious and damaging. All tenants will experience loss of service when the database is unavailable, unlike in a single-tenant application, where it brings down a single tenant and other instances remain unaffected.
